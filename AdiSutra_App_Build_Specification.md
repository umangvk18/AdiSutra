# AdiSutra — Inventory & Billing App
### Full Build Specification (Handoff Document for Claude Code)

Prepared: July 2026
Purpose: This document contains the complete, discussed-and-finalized plan for the AdiSutra saree inventory and billing app. It is meant to be handed directly to Claude Code (or any developer) as a build spec — everything from data model to screen layouts to Google API setup is covered so the build can start without re-discovering requirements from scratch.

## 1. Project Summary

**Business:** AdiSutra — a home-based saree reselling business run by the founder and their mother, sourcing handloom sarees from vendors in Kolkata and Banaras, reselling primarily to in-person/known customers in Surat (negligible online/social selling).

**Current state:** Fully manual — handwritten bills, no formal inventory tracking, no unique identification per saree.

**Goal of this app:** Replace manual tagging and handwritten billing with a simple phone-first web app that:
- Assigns a unique code + attributes to every saree as it enters stock (with a photo)
- Lets either family member create a bill by visually selecting sarees (photo-based, not code-typing)
- Automatically tracks what's sold, what's in stock, and who owes what
- Sends customers a proper bill image via WhatsApp
- Surfaces simple business intelligence: which material/region/design/vendor sells best, stock aging, profit, pending dues

**Key constraints that shaped every decision below:**
- Two users only (founder + mother) — no need for complex role management or multi-store logic
- Must be genuinely simple to operate day-to-day — big tap targets, photo-first browsing, minimal typing
- Must be free or near-free to run
- Must allow easy manual verification of raw data outside the app (hence Google Sheets as the data store, not an opaque database)
- No GST/tax invoicing needed — simple receipts are sufficient
- No physical stock-location tracking needed

## 3. Data Architecture — Google Sheets

Google Sheets was deliberately chosen over a traditional database (e.g., PostgreSQL) for this v1 build, for reasons discussed and agreed with the business owner:
- Zero cost
- Full manual visibility/editability by the owner at any time — no "black box"
- Sufficient for two-person, home-business scale usage
- Clear upgrade path to a real database later if the business scales up significantly (see Section 11)

One Google Spreadsheet (file) will contain **five sheets (tabs)**, detailed below. All ID fields should be generated by the app's backend — never left to manual entry — to guarantee uniqueness.

### 3.1 Sheet: `Inventory`

One row = one physical saree.

| Column | Type | Notes |
|---|---|---|
| `saree_code` | Text | Unique, auto-generated. Format suggestion: `AS-0001`, `AS-0002`... incrementing. Primary key for this sheet. |
| `photo_url` | Text (URL) | Link to the photo stored in Google Drive (see Section 4.3 on photo handling) |
| `region` | Text (dropdown) | Sourced from `Attributes` sheet — e.g. Kolkata, Banaras |
| `vendor` | Text (dropdown) | Sourced from `Attributes` sheet |
| `material` | Text (dropdown) | Sourced from `Attributes` sheet |
| `design_type` | Text (dropdown) | Sourced from `Attributes` sheet |
| `color` | Text (free entry, or dropdown if the owner wants to standardize later) | |
| `cost_price` | Number | What was paid to the vendor |
| `selling_price` | Number | Asking price — can be overridden at billing time (discount) |
| `date_received` | Date | |
| `status` | Text | `In Stock` / `Sold` — **never edited manually; only the app's logic changes this** (see Section 7) |
| `date_sold` | Date (nullable) | Blank while in stock; filled when sold; cleared again on full/partial return |
| `bill_number` | Text (nullable) | Blank while in stock; filled with the bill it was sold on; cleared on return |

### 3.2 Sheet: `Customers`

One row = one customer.

| Column | Type | Notes |
|---|---|---|
| `customer_id` | Text | Unique, auto-generated, e.g. `CUST-0001`. Primary key. |
| `name` | Text | |
| `phone` | Text | Stored with country code if possible (e.g. `+91XXXXXXXXXX`) — required for the WhatsApp `wa.me` link to work correctly |

Deliberately minimal. Total spend, total due, and purchase history are **never stored here** — they are always calculated live from the `Bills` and `Bill_Items` sheets (see Section 3.3 and Section 9). Storing calculated values here would risk them drifting out of sync with reality.

### 3.3 Sheet: `Bills`

One row = one bill/transaction (not one item — see `Bill_Items` for the line-item breakdown).

| Column | Type | Notes |
|---|---|---|
| `bill_number` | Text | Unique, auto-generated, e.g. `AS-B0001`. Primary key. |
| `customer_id` | Text | Foreign key → `Customers.customer_id` |
| `date` | Date | |
| `subtotal` | Number | Sum of selling prices of items on this bill, before discount |
| `discount` | Number | Manually entered at billing time |
| `total_amount` | Number | `subtotal - discount` |
| `amount_paid` | Number | Entered at billing time (full amount if "Paid Full," partial amount if "Partial," 0 if "Full Credit") |
| `amount_due` | Number | `total_amount - amount_paid` (can be computed live, but storing it as a snapshot per bill is fine since it only changes when a new payment is logged against this specific bill — see the payment-update flow in Section 7.4) |
| `payment_status` | Text | `Paid` / `Partial` / `Credit` — auto-derived from `amount_due` (0 → Paid, >0 and amount_paid>0 → Partial, amount_paid=0 → Credit) |
| `bill_status` | Text | `Active` / `Returned` / `Partially Returned` — see Section 7.5 |

### 3.4 Sheet: `Bill_Items`

One row = one saree sold on one bill (this is what makes analytics by material/region/design possible without parsing comma-separated text).

| Column | Type | Notes |
|---|---|---|
| `bill_item_id` | Text | Unique, auto-generated, e.g. `BI-00001`. Primary key. |
| `bill_number` | Text | Foreign key → `Bills.bill_number` |
| `saree_code` | Text | Foreign key → `Inventory.saree_code` |
| `price_at_sale` | Number | The selling price for this specific saree at the time of this bill (captures history even if `selling_price` in `Inventory` changes later) |
| `item_status` | Text | `Sold` / `Returned` — allows partial returns (see Section 7.5) without deleting the row (preserves history) |

### 3.5 Sheet: `Attributes`

Powers all dropdowns in the app. Lets the owner add new Region/Material/Design/Vendor values on the fly, directly from the app, without ever touching code.

| Column | Type | Notes |
|---|---|---|
| `attribute_type` | Text | One of: `Region`, `Material`, `Design`, `Vendor` |
| `value` | Text | The actual dropdown option, e.g. `Banarasi Silk` |

## 4. Technical Architecture

### 4.1 Overall shape

A single **Progressive Web App (PWA)** — a web app that can be "installed" to a phone's home screen and then looks/behaves like a native app icon, with no app-store submission process. This was chosen over a native Android/iOS app because:
- No installation friction (one link, "Add to Home Screen") — critical since one user is not tech-savvy and a Play Store install/APK-sideload process would be a barrier
- Instant updates — when the app is improved later, there's nothing for either user to manually update
- One codebase, works on both users' phones regardless of Android/iOS

### 4.2 Recommended stack

- **Frontend:** React, built with a framework that has strong PWA support out of the box (Next.js is a solid default choice — well-documented, large ecosystem, works cleanly with the hosting options below). Styling via Tailwind CSS, themed with the color palette in Section 2, for fast, consistent styling.
- **Backend logic:** Server-side API routes/serverless functions (co-located with the Next.js app, or a small separate backend — either works). **All Google API credentials must live only on the backend, never sent to the browser.** The browser should never directly hold the OAuth tokens or any secret key.
- **Data store:** Google Sheets (see Section 3)
- **File store:** Google Drive (see Section 4.3)
- **Hosting:** See Section 4.5

### 4.3 Photo handling — an important technical detail

Google Sheets cells cannot store images directly. The working pattern (as discussed and confirmed) is:

1. When a photo is taken/selected in the app, the backend **compresses and resizes it first** (e.g., max width ~1200px, JPEG quality ~80%) before uploading. This matters because: (a) it keeps uploads fast on a phone connection, and (b) it makes the free 15GB storage allowance last a very long time — a compressed saree photo at these settings is typically 150–300KB, meaning 15GB comfortably holds tens of thousands of photos, far beyond what a home business will ever need.
2. The compressed photo is uploaded to a specific Google Drive folder.
3. Drive returns a shareable link for that file.
4. That link is written into the `photo_url` column of the relevant `Inventory` row.
5. Anywhere the app needs to display the photo (the billing photo-grid, the inventory list, etc.), it simply loads the image from that stored link.

**A critical technical gotcha to flag explicitly, confirmed via research on current (2026) Google API behavior:** If file uploads are authenticated using a **service account** (a common pattern for "headless" backend-to-Google integrations), Google Drive can reject the upload with a *"Service Accounts do not have storage quota"* error. This happens because service accounts, on a standard (non-Google-Workspace) setup, do not inherit the storage quota of a regular Google account — even when uploading into a folder that's been shared with them.

**The correct fix, and what should be used for this project:** Authenticate Drive (and, for consistency, Sheets too) using **OAuth 2.0 authorized as the actual business Google account** (`adisutra507@gmail.com`), not a service account. Practically, this means:
- A one-time authorization step is performed (the business owner logs in with `adisutra507@gmail.com` and grants the app permission, exactly like a normal "Sign in with Google" flow) — this happens once, during setup, not on every app use
- The resulting **refresh token** is stored securely as a backend secret/environment variable
- The backend uses this refresh token to silently obtain fresh access tokens whenever it needs to read/write Sheets or upload to Drive — no repeated login required by either user during day-to-day use
- Because uploads are authorized *as* the real Google account, files are owned by and count against that account's normal free 15GB quota (not a broken/zero service-account quota)

This is a one-time setup nuance for whoever builds this — worth calling out clearly to Claude Code so it isn't discovered painfully mid-build.

### 4.4 API rate limits — should not be a concern at this scale

Google's current published limits for the Sheets API are on the order of hundreds of requests per 100 seconds per project, and roughly 100 requests per 100 seconds per user. For a two-person business doing at most a handful of inventory additions and bills per day, this is not a practical constraint. The one piece of good practice worth building in regardless: batch reads/writes where possible (e.g., read the whole `Inventory` sheet once for the billing photo-grid rather than one row at a time) rather than making many small individual API calls — this is both faster and naturally stays well within any limits.

### 4.5 Hosting recommendation

Both Vercel and Netlify offer workable free tiers for an app at this scale (100GB bandwidth/month and generous function-invocation allowances on either, as of current 2026 pricing — far more than a two-person business app will use). One nuance worth being transparent about: **Vercel's free "Hobby" tier terms of service technically restrict the tier to non-commercial use**, whereas **Netlify's free tier explicitly permits commercial use** within its usage limits. Since AdiSutra is a real (if small) business, **Netlify is the cleaner choice for straightforward ToS compliance**, though in practice this is a low-stakes technicality at this traffic scale either way. Cloudflare Pages is a reasonable third option if unlimited bandwidth ever becomes relevant (unlikely here).

**Recommendation: Netlify**, using its serverless Functions feature for the backend logic that talks to Google Sheets/Drive.

### 4.6 Access control

Since this app is used by exactly two people and holds business data (not handling money movement itself, just recording it), a very lightweight access gate is enough — e.g., a single shared PIN/passcode screen, or a simple login with one shared username/password. A full user-account system (sign-up flows, password resets, etc.) would be over-engineering for this use case. This should be confirmed as a preference before build (see Section 12).

## 5. Google Cloud Setup — Step-by-Step (For the Business Owner To Do)

These are the accounts/credentials that must be created before the app can function. This is real setup work on Google's side, not something the app itself can bypass — but it's a one-time process, roughly 15–20 minutes.

1. **Sign in to Google Cloud Console** (`console.cloud.google.com`) using the `adisutra507@gmail.com` account.
2. **Create a new Google Cloud project** (e.g., name it `adisutra-app`).
3. **Enable two APIs** for this project: the **Google Sheets API** and the **Google Drive API** (both found via "APIs & Services" → "Library," search and click "Enable" for each).
4. **Configure the OAuth consent screen** (under "APIs & Services" → "OAuth consent screen"): set it to "External" user type (since this isn't a Google Workspace organization), fill in basic app info (app name "AdiSutra," support email), and add the scopes for Sheets and Drive access.
5. **Create an OAuth 2.0 Client ID** (under "APIs & Services" → "Credentials" → "Create Credentials" → "OAuth client ID"), choosing "Desktop app" or "Web application" as the type depending on the exact auth flow Claude Code implements — this is a build-time decision, not something the owner needs to pre-decide.
6. **Perform the one-time authorization**: run the app's auth setup flow once, sign in as `adisutra507@gmail.com`, approve the permissions requested — this produces the refresh token mentioned in Section 4.3, which then gets stored as a secret on the hosting platform (Netlify).
7. **Create the Google Sheet** itself (in the `adisutra507@gmail.com` Google Drive), with the five tabs described in Section 3, and note its Spreadsheet ID (visible in its URL) for the backend to reference.
8. **Create a Google Drive folder** (e.g., "AdiSutra Saree Photos") in the same account, for uploaded photos to live in.

**What the business owner needs to actually do personally, summarized:**
- Have the `adisutra507@gmail.com` account ready (already have this ✅)
- Complete steps 1–5 and 7–8 above (or hand this document to Claude Code, which can walk through doing this together interactively)
- Perform the one-time "Sign in with Google" authorization in step 6 when prompted during setup

Purpose: a glanceable daily summary — nothing lives *only* here, it just surfaces highlights from the other tabs.

Purpose: everything related to stock lives here — both viewing and adding.

  - Text/number fields for Color, Cost Price, Selling Price

  - "Save Saree" button

### 6.3 Tab: Bills

Purpose: creating new bills and managing existing ones (including dues and returns).

- **Toggle at top: "New Bill" / "All Bills"**
- **New Bill flow:**
  - Photo grid of all currently **In Stock** sarees (this is the core interaction — tap-to-select, photo-first, not code-based)
  - Selected items shown with a running subtotal
  - Discount field (manual entry)
  - Customer picker: search existing customers by name/phone, or "+ New Customer" for a quick add (name + phone)
  - Payment status selection: Paid Full / Partial (enter amount) / Full Credit
  - "Confirm Bill" button — triggers the full save-and-update logic described in Section 7, then generates the bill image and opens the WhatsApp share flow (Section 8)
- **All Bills view:**
  - List of all bills, filterable by payment status (Paid / Partial / Credit) and searchable by customer
  - Pending dues prominently sortable/filterable here (this is the canonical "due list" location; Home's pending-bills section is just a shortcut into this view)
  - Tapping a bill shows its full detail (items, amounts, payment history) and offers:
    - **"Log Payment"** — for a Partial/Credit bill, record an additional payment received, updating `amount_paid`/`amount_due`/`payment_status` on that bill
    - **"Process Return"** — the dedicated, controlled return flow (see Section 7.5) rather than free-editing the bill's line items directly

Purpose: the simple CRM layer.

  - Their full purchase history (bills, computed live from `Bills`/`Bill_Items`)

  - Current amount due, if any (computed live, highlighted if greater than zero)
  - A shortcut to start a new bill for this customer, or to send a WhatsApp due-reminder (reusing the `wa.me` share pattern from Section 8)

## 7. Core Business Logic

This section describes what the app must *do automatically* — nothing here should ever require a manual sheet edit by either user.

### 7.1 Unique code generation

On "Save Saree," the backend reads the current highest `saree_code` from `Inventory`, increments it, and assigns the new code (e.g., last was `AS-0042`, new saree becomes `AS-0043`). The same incrementing pattern applies to `customer_id`, `bill_number`, and `bill_item_id` in their respective sheets. This must happen server-side, atomically enough to avoid two near-simultaneous additions colliding on the same code (a simple read-latest-then-write-next approach is sufficient at this usage scale, but the code should handle the rare case of a collision gracefully — e.g., retry with the next number if a write conflict is detected).

### 7.2 Creating a bill (the sale flow)

When "Confirm Bill" is tapped, the backend performs the following as one logical operation:
1. Create a new row in `Bills` (bill number, customer, date, subtotal, discount, total, amount paid, amount due, payment status, bill status = `Active`)
2. Create one row in `Bill_Items` per selected saree (linking that saree's code to this bill, capturing its price at time of sale)
3. For each of those sarees, update its row in `Inventory`: `status → Sold`, `date_sold → today`, `bill_number → [this bill's number]`

Neither user ever manually flips a saree's status — this is fully automatic and happens as a single reliable action the moment a bill is confirmed.

### 7.3 Generating and sending the bill (WhatsApp)

Covered in detail in Section 8.

### 7.4 Logging a later payment against a Partial/Credit bill

When "Log Payment" is used on an existing bill: the entered amount is added to that bill's `amount_paid`, `amount_due` is recalculated (`total_amount - amount_paid`), and `payment_status` is re-derived (Credit → Partial → Paid as the due amount decreases). The bill record itself is updated in place — no duplicate bill is created for a follow-up payment.

### 7.5 Returns

A **dedicated "Process Return" action** is used — bills are never freely edited line-by-line outside this controlled flow, to avoid accidental changes to legitimate past sales.

- **Full return:** the whole bill is marked `bill_status → Returned`. Every saree linked to it via `Bill_Items` has its `Inventory` row reverted: `status → In Stock`, `date_sold → cleared`, `bill_number → cleared`. The bill record itself is **kept, not deleted**, so there remains a clear audit trail of what happened.
- **Partial return:** only the specific returned saree(s) are affected. That saree's `Inventory` row reverts to In Stock as above. Its row in `Bill_Items` is flagged `item_status → Returned` (not deleted, to preserve history). The parent bill's `bill_status → Partially Returned`, and its `subtotal`/`total_amount`/`amount_due` are recalculated based on the remaining (non-returned) items.

### 7.6 Never store what can be calculated

As a general rule threaded through this design: customer total spend, customer total due, dashboard statistics, and stock-aging figures are **always computed live** from the underlying `Inventory`/`Bills`/`Bill_Items` data — never stored as separate, potentially-stale fields. This avoids the classic bug where a stored summary number silently disagrees with the transactions that produced it.

## 8. WhatsApp Bill Sending

**Chosen approach:** `https://wa.me/<phone_number>?text=<message>` deep links — not the WhatsApp Business API (which requires approval and has ongoing costs). This was a deliberate, confirmed choice given the free, frictionless nature of `wa.me`.

**Confirmed limitation and agreed handling:** `wa.me` links can only pre-fill *text*, not attach an image automatically. Since the business owner specifically wants a proper bill *image* (not just a text summary) sent to customers, the confirmed flow is:

1. On "Confirm Bill," the app generates a bill image (see Section 8.1 for the exact layout) and saves/downloads it to the phone
2. The app then opens a `wa.me/<customer's phone>?text=...` link (pre-filled with a short greeting, e.g., "Hi [Name], thank you for your purchase! Bill attached below 🌸"), using the phone number stored on that customer's record
3. WhatsApp opens directly to that customer's chat
4. The user taps the attach icon, selects the just-saved bill image (it will be at/near the top of their recent photos, since it was just generated), and taps send

This is one small extra manual step (attach) beyond a hypothetical fully-automatic send, and was explicitly agreed to be an acceptable tradeoff for getting a proper branded image instead of a plain-text bill.

## 9. Dashboard & Analytics — What's Computed, and From What

All of the following are computed live by reading `Inventory`, `Bills`, and `Bill_Items` — none of these values are stored as their own sheet rows.

| Metric | How it's computed |
|---|---|
| Today's / this month's sales | Sum of `total_amount` from `Bills` where `date` falls in the relevant range and `bill_status` isn't fully `Returned` |
| Total pending dues | Sum of `amount_due` across all `Bills` where `amount_due > 0` |
| Material-wise sales | Join `Bill_Items` → `Inventory` on `saree_code`, group by `material`, sum `price_at_sale` for items where `item_status = Sold` |
| Region-wise sales | Same join, grouped by `region` instead |
| Design-wise sales | Same join, grouped by `design_type` |
| Vendor-wise performance | Same join, grouped by `vendor` — useful for future buying decisions |
| Stock aging | From `Inventory` where `status = In Stock`, sorted by `date_received` ascending (oldest unsold stock surfaces first) |
| Profit | Per sold item: `price_at_sale - cost_price` (from the matching `Inventory` row), summed across a chosen period |
| Customer total spend / current due | Sum of `total_amount` / `amount_due` from `Bills` filtered to that `customer_id` |

## 11. Suggested Build Phases

**Phase 1 (MVP — get this working and in daily use first):**
- Google Sheets data model set up with seed Attributes data
- Google Cloud OAuth setup complete
- Add Saree flow (photo, code generation, attributes)
- Inventory list/filter view
- Create Bill flow (photo-grid selection, discount, customer picker, payment status)
- Automatic Inventory status updates on sale
- Bill image generation + WhatsApp `wa.me` share
- Customers tab (list, history, dues)
- Home tab (basic stats + pending bills list)

**Phase 2 (once Phase 1 is in real use and validated):**
- Full Dashboard tab (material/region/design/vendor analytics, stock aging, profit)
- Process Return flow (full + partial)
- Log Payment flow for existing Partial/Credit bills
- WhatsApp due-reminder shortcut from Customers tab
- Any polish: animations, refined empty states, etc.

Recommending this split so the family can start replacing the pen-and-paper workflow as early as possible, without waiting for every analytics feature to be finished first.

## 12. Open Questions to Resolve With Claude Code Before/During Build

These weren't fully settled in discussion and are worth a quick decision at build time:

1. **Access control:** shared PIN screen, or a simple single login? (Section 4.6)
2. **OAuth client type:** "Desktop app" vs "Web application" credential type in Google Cloud — this depends on exactly how Claude Code implements the one-time authorization flow; not something the owner needs to pre-decide, just flagging that a decision happens here.
3. **Discount input:** flat rupee amount, percentage, or both? (Not explicitly settled — flat amount was used in examples throughout this document, but percentage may be worth supporting too.)
4. **Customer phone number format:** should the app enforce/normalize a `+91` prefix automatically so `wa.me` links always work correctly, even if the number is typed without it?
